## 一、代码纠错

### 1. f-string 拼写错误（题98 & 修改属性二）

两段 Employee 代码中都有同一个 bug：

```python
# ❌ 你写的
print(f"{self.name}'salary is {self.salary}, and his age is {self.age}")

# ✅ 应该是
print(f"{self.name}'s salary is {self.salary}, and his age is {self.age}")
```

少了一个 `s`，`'salary` 应为 `'s salary`。

### 2. 关于 `__init__` 与属性"不可修改"的误解（重要）

你提到：

> `__init__` 两边各带两个下划线，按道理来说这个东西就是不可改的……所以 `e.age` 应该是个浅层的改变……真实的修改还是用 `hasattr` 和 `setattr`。

**这是一个重要的误解。** 双下划线（dunder）方法只是 Python 的"魔术方法"（magic method）命名约定，表示它会被 Python 在特定时机自动调用，**和"不可修改"没有任何关系**。

`e.age = 25` 是完全合法、完全"真实"的修改，它和 `setattr(e, 'age', 25)` 的效果**完全一样**，底层做的是同一件事。不存在"浅层"与"深层"的区别。`__init__` 里定义的属性和后来通过 `e.age = 25` 添加的属性，在实例的 `__dict__` 中地位完全相同：

```python
e = Employee("Tom", 5000)
e.age = 25
print(e.__dict__)  # {'name': 'Tom', 'salary': 5000, 'age': 25}
```

`hasattr` / `setattr` / `getattr` 只是提供了一种**用字符串动态访问属性**的方式，适用于属性名在运行时才能确定的场景，并不比直接赋值"更真实"。

## 二、关于实例创建时的参数传入

### 2.1 你的正确观察

你在做 Coordinate 那道题时发现，如果把 `map(int, input().split())` 直接写在构造函数参数的位置，Python 会把整个 map 对象当作**一个**参数传入，导致另一个参数缺失而报错。你由此总结出**"参数的接收要在外面做完"**，这个结论基本正确。

### 2.2 补充：`*` 解包运算符

除了你用的方式（先在外部解包），还有一种写法是用 `*` 在调用时展开：

```python
# 你的写法（完全正确）
x1, y1 = map(int, input().split())
c1 = Coordinate(x1, y1)

# 另一种等价写法
c1 = Coordinate(*map(int, input().split()))
```

`*` 会把 map 对象展开为多个位置参数，效果与先解包再传入完全一样。

### 2.3 你对"每行一个输入"情况的正确理解

你在题97中发现，当输入是**每行一个**的时候，可以直接在构造函数里写多个 `input()`：

```python
niu = Student(input(), input(), input(), input())
```

这是可行的，原因你也隐约意识到了——Python 函数参数**从左到右依次求值**，所以四个 `input()` 会按顺序依次读取四行输入。不过这种写法可读性较差，一般还是建议先读到变量里。

你同时正确地指出，**如果是同一行以空格分隔的多个值，就不能这么做**，因为 `map(...)` 或 `input().split()` 会被整体当作一个参数。这个区分是准确的。

## 三、关于属性不在 `__init__` 中定义的情况

### 3.1 你的正确观察：访问不存在的属性会报错

你在做题98时注意到，`__init__` 里没有定义 `age`，但 `printclass` 方法中引用了 `self.age`，你判断"到了这一步它应该会报错"——**完全正确**。Python 在运行到 `self.age` 时如果实例上找不到这个属性，就会抛出 `AttributeError`。

### 3.2 你的正确观察：用 `try/except` 捕获这个错误

你正确理解了 `try/except` 在这里的用途：第一次调用 `printclass()` 时 `self.age` 不存在，触发异常，被 `except` 捕获后输出 `"Error! No age"`。后来通过 `e.age = int(input())` 给实例添加了 `age` 属性，第二次调用就能正常输出了。这个理解完全正确。

### 3.3 你的正确观察：`age=None` 不会报错，但会输出 "None"

你试过在 `__init__` 中设 `self.age = None`，发现 f-string 会直接输出 `"his age is None"` 而不是报错。这是对的——`None` 是一个合法的值，f-string 会对它调用 `str()` 得到字符串 `"None"`。

你由此意识到，**题目的意图是让属性完全不存在**（而非设为 None），这样访问时才会抛出 `AttributeError`。这个判断是准确的。两种设计思路的对比：

```python
# 思路一：属性不存在 → 访问时抛 AttributeError → 用 try/except 捕获
# 思路二：属性为 None → 不会报错 → 需要用 if self.age is None 手动判断
```

## 四、关于 `hasattr` / `setattr` / `getattr`

### 4.1 你的正确理解

你准确地指出了以下几点：这三个是**内置函数**而不是方法，不是通过 `e.hasattr(...)` 调用的，而是 `hasattr(e, ...)`。参数结构方面，`hasattr` 接收两个参数（对象, 字符串形式的属性名），`setattr` 接收三个参数（对象, 字符串形式的属性名, 值）。

### 4.2 关于存储 `hasattr` 结果避免重复调用

你当时纠结的问题：

> 有没有办法把 `hasattr` 的值储存起来，不用运行两次？存到变量里的话，print 的时候怎么输出 True/False？需要转成布尔类型吗？

答案是：`hasattr` 返回的**本身就是** `bool` 类型（`True` 或 `False`），直接 `print` 就会输出 `True` 或 `False`，不需要任何转换：

```python
result = hasattr(e, 'age')
print(result)          # 直接输出 True 或 False，不需要 bool() 转换
if not result:
    setattr(e, 'age', age)
```

你存到变量里避免重复调用的思路是完全正确的，只是不需要担心输出格式的问题。

## 五、关于重载运算（题100）

### 5.1 你的正确理解

你准确地抓住了以下要点：

**关于 `__add__`**：当你写 `c1 + c2` 时，Python 会自动调用 `c1.__add__(c2)`。所以你在类里面定义好 `__add__` 之后，外部就可以直接用 `+` 号，非常直观。你的实现也是正确的——在 `__add__` 内部计算新的 x 和 y，然后 **return 一个新的 Coordinate 对象**，而不是修改自身。

**关于 `__str__`**：当你写 `print(obj)` 时，Python 会自动调用 `obj.__str__()` 拿到一个字符串，再把这个字符串打印出来。你特别强调了 `__str__` 应该 **return 一个字符串**而不是在函数内部直接 `print`，这一点非常重要，说明你确实理解了它的工作机制：

```python
# ✅ 正确：return 字符串
def __str__(self):
    return f'({self.x}, {self.y})'

# ❌ 错误：在内部 print
def __str__(self):
    print(f'({self.x}, {self.y})')  # 这样 print(c1) 会输出 None
```

### 5.2 补充："重载"这个词的含义

"重载"（overloading）在这里的意思是：Python 的 `+`、`print()` 等操作本来就有默认行为，你通过在类中定义 `__add__`、`__str__` 等魔术方法，**重新定义（覆盖）了这些操作对你的类的行为**。常见的可重载运算符还有 `__sub__`（`-`）、`__mul__`（`*`）、`__eq__`（`==`）、`__lt__`（`<`）等。

## 六、关于在 `__init__` 中做类型转换

### 你的正确做法

在题97中，你在 `__init__` 内部做了类型转换：

```python
def __init__(self, name, ide, score, grade):
    self.name = name
    self.id = ide
    self.score = int(score)       # 在 __init__ 内部转 int
    self.grade = grade.split()    # 在 __init__ 内部转 list
```

这是完全合理的做法。`__init__` 的职责就是初始化实例的状态，在里面做必要的类型转换和数据处理是很常见的模式。你提到"不知道这样可不可行"——完全可行，而且是推荐的写法。相比在外部转好再传入，把转换逻辑放在 `__init__` 里可以保证**无论谁来创建这个对象，数据格式都是统一的**。

## 九、总结

### ✅ 你的正确收获

- 创建实例时，`map()` 等返回的迭代器对象会被当作单个参数，必须先在外部解包（或用 `*` 解包）；但如果是每行一个 `input()`，可以直接写在构造函数里，Python 会从左到右依次求值。
- 访问实例上不存在的属性会抛出 `AttributeError`，可以用 `try/except` 捕获。
- 将属性设为 `None` 和属性不存在是两种不同的状态，前者不会报错但会输出 `"None"`。
- `__str__` 必须 **return 字符串**而不是在内部 print；`__add__` 应该 **return 新对象**而不是修改自身。
- `hasattr` / `setattr` 是**内置函数**而非方法，参数中属性名要用字符串形式传入。
- `hasattr` 的结果可以存到变量里复用，直接 print 就能输出 `True`/`False`。
- 在 `__init__` 内部做 `int()` 或 `.split()` 等类型转换是合理且推荐的做法。
- 递归函数需要基准情形和递推关系两部分。

### ❌ 需要修正的认知

- **双下划线方法和"不可修改"无关。** `__init__` 只是自动在实例化时被调用的初始化方法，不代表它定义的属性有什么特殊保护。
- **`e.age = 25` 和 `setattr(e, 'age', 25)` 完全等价，** 都是真实的属性修改，不存在"浅层"与"深层"的区别。`setattr` 的优势仅在于可以用字符串动态指定属性名。
